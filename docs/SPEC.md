## 2. アプリの目的

「東北クマ出没・安全確認マップ」は、東北地方のクマ出没・目撃情報を地図と一覧で確認できるようにし、地域住民・旅行者・登山者などの安全確認を支援するためのアプリである。

公式情報をもとに、利用者が危険エリアや最新状況を把握しやすくすることを目的とする。

### 主な目的

* 東北地方のクマ出没・目撃情報を地図上で確認できるようにする
* 各県・自治体などの公式情報に基づいた情報を掲載する
* 住民・旅行者・登山者・釣り人・キャンパーなどの安全確認を支援する
* 最新の出没状況を一覧で確認できるようにする
* 出没場所の詳細住所ではなく、概略位置として安全配慮した表示を行う
* 情報源URLを明示し、利用者が一次情報へアクセスできるようにする

---

## 3. 想定ユーザー

### 主要ユーザー

* 東北地方に住む人
* 通勤・通学・買い物などで屋外移動する人
* 登山・釣り・キャンプ・観光で東北を訪れる人
* 家族や高齢者の安全を確認したい人
* 子どもやペットの外出時に地域の安全を確認したい人

### 利用シーン

* 外出前に近隣の出没情報を確認する
* 登山・キャンプ・釣りの前に目的地周辺の情報を確認する
* 家族に注意喚起するために情報を共有する
* 旅行前に訪問エリアの安全状況を確認する
* 自治体公式情報への導線として利用する

---

## 4. 解決したい課題

### 現状の課題

* クマ出没情報が県・市町村・警察・報道などに分散している
* 地図で直感的に確認しにくい
* 最新情報と過去情報の区別が分かりにくい
* 旅行者や登山者が地域別に安全確認しにくい
* 公式情報へのリンクがまとまっていない
* SNSや非公式情報だけでは信頼性の判断が難しい

### アプリで解決すること

* 東北地方の出没情報を1か所で確認できるようにする
* 地図と一覧の両方で確認できるようにする
* 情報源URLを表示し、公式情報にアクセスできるようにする
* 最終更新日を明示する
* 出没日・場所・内容・情報源を分かりやすく表示する
* 危険を断定せず、安全確認を補助する情報として提供する

## 7. 画面構成

### 7.1 トップページ `/`

#### 目的

アプリの概要を伝え、マップ・更新情報へ誘導する。

#### 表示内容

* アプリ名
* アプリの簡単な説明
* 最終更新日
* マップを見るボタン
* 最新情報を見る導線
* 注意事項・免責の概要
* 公式情報をもとにしている旨
* 東北6県を対象としている旨

#### 主なCTA

* 「地図で確認する」
* 「更新履歴を見る」
* 「注意事項を確認する」

---

### 7.2 マップページ `/map`

#### 目的

クマ出没・目撃情報を地図と一覧で確認できるようにする。

#### 表示内容

* 地図
* 出没地点マーカー
* 一覧表示
* 都道府県
* 市町村
* 発生日
* 内容
* 情報源URL
* 最終更新日
* データ件数
* 注意事項

#### マーカー表示

* CSVに緯度・経度がある情報のみ地図上に表示する
* 緯度・経度がない情報は一覧には表示するが、地図マーカーは表示しない
* マーカー位置は概略位置とし、詳細地点の特定を避ける
* クリックまたはタップで概要を表示する

---

### 7.3 更新履歴ページ `/updates`

#### 目的

アプリやデータの更新履歴を確認できるようにする。

#### 表示内容

* 更新日
* 更新内容
* 対象県
* 反映内容の概要
* データ更新か、サイト修正かの区別

#### 表示例

| 日付         | 種別    | 内容              |
| ---------- | ----- | --------------- |
| 2026-06-20 | データ更新 | 秋田県・岩手県の最新情報を反映 |
| 2026-06-20 | サイト更新 | 最終更新日表示を更新      |

---

### 7.4 注意事項・免責エリア

#### 目的

利用者に対し、アプリの位置づけと限界を明確に伝える。

#### 表示内容

* 本アプリは安全確認を補助するための情報提供である
* 公式発表・自治体・警察等の情報を必ず確認すること
* 掲載情報は最新性・完全性を保証しない
* クマを見かけた場合は近づかず、自治体・警察等の案内に従うこと
* 緊急時は各地域の緊急連絡先へ連絡すること

## 8. ユーザー導線

### 基本導線

1. ユーザーがトップページにアクセスする
2. アプリの目的と最終更新日を確認する
3. 「地図で確認する」から `/map` に移動する
4. 地図または一覧で出没情報を確認する
5. 必要に応じて情報源URLから公式情報を確認する
6. 更新履歴を見たい場合は `/updates` に移動する

### 外出前の導線

1. トップページにアクセス
2. 最終更新日を確認
3. `/map` で目的地周辺の情報を確認
4. 一覧で発生日・市町村・内容を確認
5. 公式情報源を開いて詳細確認
6. 外出判断の参考にする

### 旅行者・登山者の導線

1. `/map` にアクセス
2. 訪問予定エリア周辺を確認
3. 直近の出没情報を確認
4. 情報源URLで自治体・警察等の一次情報を確認
5. 必要に応じて予定変更・装備確認を行う

## 9. 入力項目

### 9.1 必須項目

| 項目名           | 型      | 必須 | 説明             |
| ------------- | ------ | -: | -------------- |
| id            | string | 必須 | 各情報の一意ID       |
| prefecture    | string | 必須 | 都道府県名          |
| municipality  | string | 必須 | 市町村名           |
| occurred_date | string | 必須 | 出没・目撃などが発生した日付 |
| title         | string | 必須 | 情報の短い見出し       |
| description   | string | 必須 | 内容の概要          |
| source_name   | string | 必須 | 情報源名           |
| source_url    | string | 必須 | 情報源URL         |
| updated_at    | string | 必須 | アプリ側で反映した日付    |

---

### 9.2 任意項目

| 項目名                   | 型       | 必須 | 説明              |
| --------------------- | ------- | -: | --------------- |
| latitude              | number  | 任意 | 緯度。概略位置で管理      |
| longitude             | number  | 任意 | 経度。概略位置で管理      |
| location_label        | string  | 任意 | 表示用の概略地点名       |
| category              | string  | 任意 | 目撃・出没・痕跡・人身被害など |
| time_of_day           | string  | 任意 | 朝・昼・夕方・夜など      |
| status                | string  | 任意 | 確認済み・要確認・情報不足など |
| note                  | string  | 任意 | 補足事項            |
| source_published_date | string  | 任意 | 情報源が公開した日付      |
| display               | boolean | 任意 | 表示対象にするかどうか     |

---

### 9.3 CSV項目の例

## 11. 判定項目

### 11.1 表示対象判定

CSV内の各行について、以下を満たす場合に表示対象とする。

| 判定項目          | 条件            |
| ------------- | ------------- |
| display       | `true` または未指定 |
| prefecture    | 東北6県のいずれか     |
| occurred_date | 日付として解釈可能     |
| title         | 空でない          |
| description   | 空でない          |
| source_url    | 空でない          |
| source_name   | 空でない          |

---

### 11.2 地図表示判定

以下を満たす場合に地図マーカーを表示する。

| 判定項目                 | 条件              |
| -------------------- | --------------- |
| latitude             | 数値である           |
| longitude            | 数値である           |
| latitude / longitude | 東北地方として不自然でない範囲 |
| display              | 表示対象である         |
| location_label       | 可能な限り設定されている    |

緯度・経度がない場合は、地図には表示せず、一覧のみに表示する。

---

### 11.3 最新情報判定

最新情報として強調するかどうかは、`occurred_date` または `source_published_date` をもとに判定する。

#### 仮定

* 直近7日以内の情報を「新しい情報」として扱う
* 直近30日以内の情報を「最近の情報」として扱う
* 30日を超える情報は通常表示とする

| 判定    | 条件              |
| ----- | --------------- |
| 新しい情報 | 発生日または公開日が7日以内  |
| 最近の情報 | 発生日または公開日が30日以内 |
| 通常情報  | 30日超            |

※ この判定は危険度を示すものではなく、情報の新しさを示すためのものとする。

---

### 11.4 情報源判定

| 判定    | 条件                      |
| ----- | ----------------------- |
| 公式情報  | 県・市町村・警察・公的機関等のURL      |
| 要確認   | 情報源URLはあるが、公式性の判断が必要    |
| 掲載対象外 | 情報源URLがない、または信頼性を確認できない |

MVPでは、原則として公式情報または公的性質の高い情報のみを掲載する。

---

### 11.5 危険度判定

MVPでは、独自の危険度スコアは表示しない。

理由：

* クマの行動範囲や状況は専門的判断が必要である
* アプリ側で「安全」「危険なし」と断定することは不適切である
* 利用者の誤解を避ける必要がある
* 公式情報の補助として提供する位置づけを守るため

表示する場合は、危険度ではなく以下のような表現に留める。

* 新しい情報
* 最近の情報
* 公式情報あり
* 位置は概略表示
* 詳細は公式情報を確認

## 12. 基本ロジック

### 12.1 データ読み込み

1. CSVファイルを読み込む
2. 各行をオブジェクト化する
3. 必須項目を検証する
4. 表示対象の行のみ抽出する
5. 日付の新しい順に並べる
6. `/map` と一覧に表示する

---

### 12.2 並び順

原則として以下の順に表示する。

1. 発生日が新しいもの
2. 情報源公開日が新しいもの
3. アプリ反映日が新しいもの
4. idの降順

---

### 12.3 フィルタリング

MVPでは必須ではないが、将来的に以下のフィルタを想定する。

* 都道府県
* 市町村
* 発生日
* 種別
* 新しい情報のみ
* 地図表示可能な情報のみ

MVP段階では、フィルタ機能が未実装でもよい。

---

### 12.4 地図表示ロジック

1. 表示対象データを取得する
2. 緯度・経度があるデータのみマーカー対象にする
3. マーカーを地図上に表示する
4. マーカータップ時に概要カードを表示する
5. 公式情報URLへのリンクを表示する

---

### 12.5 一覧表示ロジック

1. 表示対象データを取得する
2. 発生日の新しい順に並べる
3. 各情報をカード形式またはテーブル形式で表示する
4. 情報源URLを外部リンクとして表示する
5. 緯度・経度がない情報も一覧には表示する

---

### 12.6 最終更新日ロジック

最終更新日は、以下のいずれかで管理する。

#### 案A：手動更新

* サイト内に固定値として記載
* 保守作業時に手動で更新する

#### 案B：CSVの `updated_at` 最大値から自動算出

* CSV内の `updated_at` の最大日付を最終更新日として表示する

#### MVP推奨

MVPでは、運用ミスを減らすため、可能であれば **CSVの `updated_at` 最大値から自動算出** する。

## 13. エラー/例外時の扱い

### 13.1 CSV読み込み失敗

#### 状態

CSVファイルが読み込めない。

#### 表示

* 「データを読み込めませんでした」
* 「時間をおいて再度アクセスしてください」
* 可能であればトップページへのリンクを表示する

#### 内部処理

* 画面全体をクラッシュさせない
* エラー内容を開発者が確認できる形にする
* 利用者には技術的な詳細を表示しない

---

### 13.2 データが0件の場合

#### 状態

表示対象データが存在しない。

#### 表示

* 「現在表示できる情報はありません」
* 「最新情報は各自治体・警察等の公式情報をご確認ください」

---

### 13.3 緯度・経度がない場合

#### 状態

CSVに緯度・経度がない。

#### 表示

* 地図には表示しない
* 一覧には表示する
* 「地図表示なし」または「位置情報未設定」と表示してもよい

---

### 13.4 情報源URLがない場合

#### 状態

`source_url` が空である。

#### 扱い

* 原則として表示対象外
* どうしても掲載が必要な場合は、`status` を `要確認` とする
* MVPでは掲載しない方針を推奨する

---

### 13.5 日付が不正な場合

#### 状態

`occurred_date` が日付として解釈できない。

#### 扱い

* 原則として表示対象外
* 開発時に検出できるようにする
* 可能であればコンソールまたはログに警告を出す

---

### 13.6 外部リンク切れ

#### 状態

情報源URLが無効、またはリンク切れ。

#### 扱い

* 表示自体は維持してもよい
* 定期保守時に確認する
* リンク切れが確認された場合は、代替URLまたは公式トップページへの変更を検討する

---

## 15. 注意事項

### 情報の扱い

* 本アプリは公式情報を補助的に見やすく整理するものである
* 掲載情報の完全性・即時性は保証しない
* 安全判断は、必ず自治体・警察・施設管理者などの最新情報を確認したうえで行うこと
* クマを目撃した場合は、近づかず、自治体・警察等の案内に従うこと

---

### 位置情報の扱い

* 詳細住所や個人宅が特定される表示は避ける
* 緯度・経度は概略位置として扱う
* 学校・住宅・個人施設などが特定される可能性がある場合は、位置をぼかす
* 情報源に詳細住所があっても、アプリ上では必要以上に詳細表示しない

---

### 表現上の注意

使用してよい表現：

* 「クマの目撃情報」
* 「出没情報」
* 「公式情報を確認してください」
* 「安全確認の参考情報」
* 「位置は概略表示です」
* 「最新情報は情報源をご確認ください」

避ける表現：

* 「この地域は安全です」
* 「この地域は危険です」
* 「必ず出没します」
* 「絶対に安全」
* 「危険度○○%」
* 「ここに行ってはいけません」
* 「公式情報より正確です」

---

### 法的・倫理的配慮

* 誤解を招く危険度表示は避ける
* 個人や施設を特定できる情報は表示しない
* 公式情報源を明示する
* 非公式情報を扱う場合は、信頼性の確認ルールを別途定める
* 緊急時の判断をアプリに依存させない

---

* 自動収集をMVPで実装する

## 18. Codex実装依頼時の参照ルール

### Codexへの依頼時に守ること

* 本仕様に反する実装は行わない
* MVPでやらないことは実装しない

* 既存アプリの構成を大きく壊さない
* CSVデータ構造に変更がある場合は、理由を明記する
* 画面表示を変更する場合は、スマホ表示を優先する
* 安全性を断定する文言を追加しない
* 公式情報URLの表示を維持する
* 位置情報は概略表示を前提にする
* 実装後は `/map` と `/updates` を確認する

## 19. 完了条件

本仕様に基づくMVP実装または修正の完了条件は以下とする。

* トップページが表示できる
* `/map` が表示できる
* CSVデータが読み込まれる
* 表示対象データが一覧に表示される
* 緯度・経度があるデータが地図に表示される
* 情報源URLが表示される
* 最終更新日が表示される
* `/updates` が表示できる
* データが0件でも画面が壊れない
* CSV読み込み失敗時にエラー表示が出る
* スマートフォンで大きく崩れない
* 安全性を断定する表現がない
* 詳細住所や個人特定につながる表示がない

## 20. 備考

本アプリは、東北地方のクマ出没・目撃情報を分かりやすく確認するための安全確認支援アプリである。

ただし、災害・野生動物・地域安全に関わる情報は常に変化するため、本アプリのみで判断せず、必ず自治体・警察・施設管理者などの最新公式情報を確認すること。

今後の機能追加・画面変更・データ仕様変更は、この `SPEC.md` を更新したうえで行う。
