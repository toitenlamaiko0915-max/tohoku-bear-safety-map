## 4. 検収ルール

### 4.1 検収の基本手順

Codexの実装後は、以下の順番で確認する。

1. Codexの報告内容を読む
2. 変更されたファイルを確認する
3. 依頼内容と変更内容が一致しているか確認する
4. ローカルまたはPreview URLで画面を確認する
5. 主要ページの表示・動作を確認する
6. 既存機能に影響がないか確認する
7. 問題があればバグ記録に追記する
8. 合格 / 要修正 / 保留を判定する
9. 必要に応じてCodexへの修正依頼文を作成する
10. 修正後に再テストする

### 4.2 検収時に見る場所

| 確認場所                | 確認内容                |
| ------------------- | ------------------- |
| Codexの報告            | 何を変更したか、動作確認したか     |
| GitHub Pull Request | 変更ファイル、差分、余計な変更の有無  |
| Files changed       | CSV、画面、ロジック、文言の変更内容 |
| Vercel Preview      | 実装後の表示確認            |

## 8. 合格 / 要修正 / 保留の判定基準

### 8.1 合格

以下をすべて満たす場合は「合格」とする。

* 依頼内容を満たしている
* 余計な変更が入っていない
* 既存機能が壊れていない
* SPEC.mdと矛盾していない
* 主要ページが表示できる
* データ表示に明らかな問題がない
* Codexの報告に変更内容と確認方法が書かれている
* 次に対応すべき大きな問題がない

### 8.2 要修正

以下のいずれかに該当する場合は「要修正」とする。

* 依頼内容を満たしていない
* 依頼していない余計な変更が入っている
* 既存機能が壊れている
* 地図や一覧が表示されない
* CSVデータが正しく反映されていない
* 公式情報URLではないURLが使われている
* SPEC.mdと矛盾している
* ビルドエラーがある
* 利用者に誤解を与える表示がある
* 安全確認アプリとして不適切な表示がある

### 8.3 保留

以下の場合は「保留」とする。

* 画面確認がまだできていない
* Preview URLが確認できない
* Codexの報告だけでは判断できない
* 公式情報との照合が未完了
* 仕様確認が必要
* 追加確認後に判断すべき内容がある
