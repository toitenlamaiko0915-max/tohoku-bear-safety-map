# CODEX_PROMPTS.md

## プロジェクト名

東北クマ出没・安全確認マップ

## プロジェクトコード

TBM

## このファイルの目的

このファイルは、東北クマ出没・安全確認マップにおいて、Codexへ依頼した作業内容をチケット番号つきで履歴管理するための管理ファイルです。

目的は、以下の通りです。

* どの作業をいつCodexへ依頼したかを記録する
* 作業内容・対象範囲・完了条件を明確にする
* 仕様相談・実装・修正・検収・デプロイ準備を混ぜない
* ChatGPTとCodexの作業履歴を追いやすくする
* 後から見返したときに、開発の流れを迷わず確認できるようにする

## アプリ概要

### アプリ名

東北クマ出没・安全確認マップ

### アプリの目的

東北地方のクマ出没・目撃情報を地図と一覧で確認できるようにし、地域住民・旅行者・登山者などの安全確認を支援するためのアプリ。

公式情報をもとに、利用者が危険エリアや最新状況を把握しやすくする。

### 想定ユーザー

* 東北地方に住む人
* 通勤・通学・買い物で屋外移動する人
* 登山・釣り・キャンプ・観光で東北を訪れる人
* 家族や高齢者の安全を確認したい人

### 技術スタック

* Next.js
* React
* TypeScript
* Vercel
* CSVデータ管理

### 現在の状態

既存アプリあり・公開済み・一部運用中

# チケット管理ルール

## チケット形式

```text
TBM-001｜作業名
```

例：

```text
TBM-001｜トップページに最終更新日を表示
TBM-002｜CSVデータの最新情報更新
TBM-003｜/updatesページに更新履歴を追加
```

---

## 採番ルール

* チケット番号は3桁で管理する
* 形式は `TBM-001｜作業名` とする
* 1つのCodex依頼につき1チケットを作成する
* 仕様相談、実装、修正、検収、デプロイ準備を1つのチケットに混ぜない
* Codex指示文のタイトルには必ずチケット番号を入れる
* 次の番号が不明な場合は、`PROJECT_LOG.md` の「最新のCodex依頼番号」を確認する
* 作業が完了したら、このファイルの「依頼履歴」に追記する
* 未実施の案は「未実施チケット候補」に置き、正式依頼時にチケット番号を付ける

---

## チケットの粒度

チケットは、Codexが迷わず作業できる単位に分ける。

### 良い粒度

* 1つの画面に1つの表示を追加する
* CSVデータの更新のみを依頼する
* `/updates` ページに更新履歴を追加する
* 表示崩れの修正のみを依頼する
* PR作成前の確認項目を整理する

### 避ける粒度

* 複数ページ・複数機能・デプロイ作業をまとめて依頼する
* 仕様整理と実装を同じ依頼に入れる
* データ更新とデザイン変更を同時に依頼する
* 検収と追加修正を同じチケットにする
* 「いい感じに直して」など、作業範囲が曖昧な依頼にする

- 対象ページ：
- 対象ファイル：
- 対象データ：
- 対象機能：

## 依頼内容

1. 〇〇を確認する

4. 〇〇をテストする

## やってほしいこと

- 〇〇
- 〇〇
- 〇〇

## やらないでほしいこと

- 関係ないファイルを変更しない
- 既存デザインを大きく変更しない
- 推測でデータを追加しない
- 公式情報が確認できない内容を断定しない
- デプロイ作業は行わない
- PRのマージは行わない

## データ・情報源の扱い

- クマ出没・目撃情報は、原則として公式情報をもとにする
- source_urlには公式情報のURLを入れる
- 情報源が確認できない場合は、推測で補完しない
- 位置情報は詳細住所ではなく、概略表示を基本とする
- 個人が特定される情報は追加しない

## テスト条件

以下を確認してください。

- 対象ページが正常に表示されること
- 表示崩れがないこと
- TypeScriptエラーがないこと
- 既存機能に影響がないこと
- CSVを変更した場合、地図と一覧に反映されること
- source_urlが正しくリンクとして機能すること

## 完了条件

- 依頼内容がすべて反映されていること
- 変更ファイルが必要最小限であること
- 変更内容の概要を報告すること
- テスト結果を報告すること
- PRを作成できる状態になっていること

# Codexへの禁止事項

Codexへ依頼するときは、原則として以下を禁止事項に含める。

## 共通の禁止事項

* 関係ないファイルを変更しない
* 既存デザインを大きく変更しない
* 既存の画面構成を勝手に変更しない
* 既存機能を削除しない
* 仕様にない機能を追加しない
* 不要なリファクタリングを行わない
* 推測でデータを追加しない
* 公式情報が確認できない内容を断定しない
* 個人情報や詳細住所を追加しない
* PRのマージは行わない
* 本番環境へのデプロイは行わない

## データ更新時の禁止事項

* 非公式情報だけを根拠にCSVへ追加しない
* source_urlを空欄にしない
* 位置情報を詳細にしすぎない
* 個人宅・学校・施設などが特定される粒度で表示しない
* 確認できない日付を入れない
* 既存CSVの形式を勝手に変更しない

## UI修正時の禁止事項

* 全体デザインを大きく変えない
* 配色・余白・フォントを大幅に変更しない
* スマホ表示を崩さない
* 地図表示や一覧表示の既存仕様を壊さない

# 未実施チケット候補

正式なチケット番号を付ける前の候補をここに置く。
Codexへ依頼する段階で、`TBM-001` から順番に正式採番する。

| 候補名              | 種別    | 優先度 | 概要                     | 備考                 |
| ---------------- | ----- | --: | ---------------------- | ------------------ |
| 最新情報をCSVへ反映      | データ更新 |   高 | 各県公式の最新情報を確認し、CSVへ反映する | source_urlは公式情報にする |
| サイト最終更新日を更新      | 運用更新  |   高 | トップページ等の最終更新日を最新にする    | 更新日ルールを確認          |
| /updatesに更新履歴を追加 | 運用更新  |   中 | データ更新履歴を一覧で確認できるようにする  | 保守履歴管理用            |
| 表示崩れ確認           | 検収    |   中 | スマホ・PCで地図と一覧の表示を確認する   | 実装後の確認用            |
| CSV項目の整理         | 仕様整理  |   低 | CSVの項目名・入力ルールを整理する     | 実装とは分ける            |

# 更新ルール

このファイルは、Codexへ依頼を出すたびに更新する。

## 依頼前

* チケット番号を確認する
* 作業内容を1チケットに絞る
* Codex依頼文のタイトルにチケット番号を入れる
* 禁止事項を明記する
* テスト条件と完了条件を明記する

## 依頼後

* 依頼履歴にチケットを追記する
* 状態を更新する
* Codexからの報告内容を確認する
* 必要に応じて関連PRを記録する

## 状態の種類

| 状態    | 意味              |
| ----- | --------------- |
| 未着手   | まだCodexへ依頼していない |
| 依頼中   | Codexへ依頼済みで作業中  |
| PR確認中 | PRが作成され、内容確認中   |
| 修正依頼中 | Codexへ追加修正を依頼中  |
| 完了    | マージまたは作業完了済み    |
| 保留    | 今は対応しない         |
| 中止    | 実施しないことにした      |

# 注意事項

* このファイルは、Codex依頼の履歴管理用である
* 詳細仕様は `SPEC.md` を確認する
* 現在地や進行状況は `PROJECT_LOG.md` を確認する
* Codexへ依頼する前に、作業内容が1チケットに収まっているか確認する
* 仕様変更がある場合は、実装依頼の前に仕様整理チケットを作成する
* デプロイやマージはCodexに任せず、人間側で確認してから行う
* 安全情報を扱うため、公式情報・概略表示・個人情報保護を優先する
